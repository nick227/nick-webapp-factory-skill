# Nick Web App Factory

You are building User's web apps using his preferred stack and architecture.

**Your primary directive: keep building until the work is done. Do not stop at plans, outlines, or summaries. Write real files. Generate real code. Keep going until the app runs.**

---

## Bias Toward Action

**This applies once an MVP definition exists.** Before that, run Phase 0. After it, charge forward without asking permission.

- If you have enough context to generate a file, generate it. Don't ask for permission.
- If a step is ambiguous, make a reasonable decision, note it inline with a comment, and keep going.
- If you finish one layer (e.g. schema), immediately move to the next (routes, then frontend pages, then components).
- Never stop at a "here's the plan" summary. Plans are only valid if followed immediately by implementation.
- After completing a section, say what you're doing next and do it — do not ask "should I continue?"
- If context runs long, summarize what's been built and list exactly what remains, then continue.

## Default Stack

Unless overridden:

| Layer | Choice |
|---|---|
| Monorepo | PNPM workspaces |
| Web app | `apps/web` — Vite + React + TypeScript + Tailwind |
| API server | `apps/server` — Fastify + TypeScript |
| Database layer | `packages/db` — Prisma + MySQL |
| API contract | `packages/api-spec` — OpenAPI 3.1 YAML, **single source of truth for all routes** |
| Generated types | `packages/sdk/src/generated/` — auto-generated from OpenAPI via `openapi-typescript` |
| API client | `packages/sdk/src/client.ts` — `openapi-fetch` client, fully typed from generated types, no hand-written methods |
| Hook layer | `packages/sdk/src/hooks/` — framework-agnostic React Query hooks, **portable across any React app** |
| Testing | Vitest |
| Linting | ESLint + Prettier |
| Type checking | `tsc --noEmit` |

## The OpenAPI → SDK Pipeline

This is the architectural spine. Everything flows from the OpenAPI spec.

```
packages/api-spec/openapi.yaml        ← you write this (or generate from Fastify routes)
        │
        ├──────────────────────────────────────────────────────┐
        ▼                                                      ▼
packages/sdk/src/generated/types.ts   ← openapi-typescript    apps/server/src/__tests__/
        │                                                      ← test stubs generated by
        ▼                                                        scripts/generate-tests.ts
packages/sdk/src/client.ts            ← openapi-fetch client
        │
        ▼
packages/sdk/src/hooks/               ← React Query hooks (portable, no app coupling)
        │
        ▼
apps/web/src/pages/                   ← pages import hooks from @project/sdk, nothing else
```

**Rules:**
- The OpenAPI YAML is written first. No route exists in Fastify that isn't in the spec.
- `openapi-typescript` generates `types.ts` — never write these types by hand.
- The client in `packages/sdk/src/client.ts` uses generated types for all request/response shapes.
- Hooks live in `packages/sdk` — never in `apps/web`. A hook that lives in `apps/web` is a bug.
- A new frontend (React Native, Next.js, etc.) gets the full API + hook layer by installing `@project/sdk`.
- Never import from `apps/server` in any frontend code. The SDK is the only bridge.

## Architecture Rules

- **OpenAPI is the contract.** The spec is written before implementation. Fastify implements what the spec declares. The SDK consumes what the spec declares. They must match — the spec is the referee.
- **The spec drives wiring on both sides.** `fastify-openapi-glue` reads the spec to route requests and validate them server-side. `openapi-fetch` + generated types give the client fully typed path/param/response inference. Neither side requires hand-written glue code — the spec does it.
- **SDK is the only frontend-to-backend bridge.** Pages and components never call `fetch` or `axios` directly. All data access goes through hooks in `@project/sdk`; streaming plugin hooks may use `fetch` internally when documented in `references/plugin-guide.md`.
- **Data hooks live in the SDK, not the app.** If a hook fetches or mutates data, it belongs in `packages/sdk/src/hooks/` — never in the app. Non-data utility hooks (`useDebounce`, `useLocalStorage`, `useIntersectionObserver`) live in `apps/web/src/lib/`. A `hooks/` folder inside `apps/web/` should not exist — it signals data-fetching code leaking out of the SDK.
- **SDK hooks are framework-portable.** They use React Query (`@tanstack/react-query`) which works in Vite, Next.js, Expo, etc. No Vite-specific imports, no router coupling, no app-level context except `QueryClient`.
- **Generated types are never edited by hand.** If a type is wrong, fix the OpenAPI spec and regenerate.
- **No route registration by hand.** `fastify-openapi-glue` maps `operationId` → handler at startup. A missing export is a startup error. No `fastify.get/post/patch/delete()` calls anywhere in the server.
- **No client methods written by hand.** `openapi-fetch` exposes `client.GET/POST/PATCH/DELETE` typed directly from the spec. Adding a route to the spec makes it available to hooks immediately — no `client.ts` changes needed.
- **Test structure is spec-driven.** `pnpm test:generate` produces one test stub per `operationId`. Auth tests are generated automatically based on the route's `security` declaration. Response shape is validated against the spec schema via Ajv in every test. Developers only add seed data and business-rule assertions.
- **Frontend is data-agnostic.** Pages compose components and call hooks. Components receive typed props. Neither knows about Prisma, Fastify, or MySQL.
- **API owns business rules.** Frontend never recomputes things the backend should return.
- **Prisma is the DB source of truth.** Never define DB structure outside `schema.prisma`.
- **Backend naming is serious.** Product copy can be branded. Model/route names stay plain.
- **Docs are generated where possible.** `pnpm docs:generate` produces `docs/api-reference.md`, `docs/env-vars.md`, and `docs/database.md` from living artifacts. Narrative docs live in `docs/` and are committed alongside code.
- **Admin is not a shortcut.** Admin capabilities go through the full pipeline: schema → spec → SDK regenerate → handlers → tests → frontend. `adminAuth` is a security handler variant, not a flag checked inside routes. Admin belongs in Phase 4, not Phase 6.
- **`CLAUDE.md` is the project memory.** Generated at the end of Phase 5 and updated through Phase 6. Future sessions read it first to understand what's been built, which phase was last completed, and any deviations from defaults.
- **Templates-first.** Before generating any component, layout, config, or script, check `templates/` in the skill folder for a ready-made implementation. Copy the template verbatim into the project and adapt only what differs (nav items, route names, paths). Generating from scratch when a template exists is a bug — it introduces variance and wastes tokens. See `references/registry.md` for the full template index and phase-to-template mapping.

## Social App Defaults

When the app has social features:

- `User` and `Profile` are separate models.
- Follows/friendships are explicit join models (never just an array field).
- Feed is cursor-paginated from the start.
- Notifications are derived from events, not written ad hoc.
- Media model is abstract enough to support image, audio, and video later.

## MVP Discipline

Always separate:

- **MVP** — what ships first
- **Phase 2** — what comes next
- **Parking lot** — ideas that don't belong yet

Do not implement Phase 2 items during MVP build. Add a `## Phase 2` comment in relevant files to mark deferred work.

---

## Code Style

- Use `async/await`, not `.then()` chains.
- Fastify handlers delegate to service functions. No business logic in route handlers.
- React components are functional. Hooks for all stateful logic.
- Tailwind for all styles — no CSS files unless unavoidable.
- Error responses follow a consistent shape: `{ error: string, code?: string }`.
- Success responses: `{ data: T }` wrapper for single resources, `{ data: T[], meta: { hasMore: boolean, nextCursor: string | null } }` for paginated lists.
